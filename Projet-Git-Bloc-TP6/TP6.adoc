== 1. Comparaison : Reflected vs Stored (Lab 1 & 2)

*Question : Quelle est la différence majeure de dangerosité ? Quelle méthode est la plus efficace pour piéger un administrateur sans soupçons ?*

La différence réside principalement dans la **durée** et la **portée** de l'attaque :

* **Reflected (Lab 1)** : Le code malveillant ne fonctionne que pour la personne qui clique sur un lien précis que vous lui envoyez. S'il ne clique pas, rien ne se passe.
* **Stored (Lab 2)** : Le code est enregistré directement sur le site. Il s'exécute automatiquement pour chaque personne qui visite la page, sans avoir besoin de leur envoyer de lien.

[TIP]
====
**La méthode la plus efficace :** Pour piéger un administrateur, le **Stored XSS (Lab 2)** est idéal. Le code étant intégré au site, l'administrateur le déclenche simplement en consultant la page normalement, ce qui ne génère aucun soupçon.
====

== 2. Contexte d’attribut (Lab 3)

*Question : Pourquoi <script>alert(1)</script> échoue-t-il ici ?*

* **Blocage serveur** : Le site ne laisse pas passer les signes `<` et `>`. Le serveur les transforme en codes de texte inoffensifs (encodage).
* **Réaction du navigateur** : Le navigateur reçoit ces signes modifiés et les traite comme du texte simple à afficher, et non comme une instruction.
* **Résultat technique** : Puisqu'il ne reconnaît pas de balise ouvrante ou fermante, le moteur de script n'est pas activé. Le code reste "inerte". On cherche alors à injecter une instruction dans une balise existante.

== 3. Syntaxe JavaScript (Lab 4)

*Question : Quel est le rôle de chaque partie du payload `'; alert(1); //` ?*

Lors d'une injection dans un script existant, chaque symbole permet de détourner le code sans l'interrompre :

* ` ' ` **(Guillemet simple)** : Sert à fermer la zone de texte commencée par le site pour transformer la suite en instruction.
* ` ; ` **(Point-virgule)** : Sépare deux ordres différents. Il finit la première instruction pour lancer l'ordre suivant.
* ` // ` **(Doubles barres)** : Annule tout ce qui suit sur la même ligne en le transformant en commentaire invisible, évitant ainsi une erreur de lecture.

== 4. Cas du DOM XSS et Pare-feu (Lab 5)

*Question : Pourquoi un pare-feu (WAF) a-t-il du mal à détecter cette faille ?*



* **Lieu de l'exécution** : Contrairement aux failles classiques, le code malveillant reste exclusivement dans votre navigateur. C'est votre propre ordinateur qui lit l'adresse de la page et décide d'écrire le code sur l'écran.
* **Invisibilité pour le WAF** : Le serveur ne reçoit jamais la partie de l'adresse contenant l'attaque.
* **Conclusion** : Le pare-feu examine ce qui entre et sort du serveur. Comme l'attaque se déroule entièrement dans la navigation locale sans franchir la porte du serveur, le pare-feu n'a aucune information à vérifier.

== 5. Remédiation : La règle d’or

*Question : Si vous étiez développeur, quelle solution choisiriez-vous ?*

La solution absolue est d'**encoder (échapper) toutes les données affichées à l'écran**.

[cols="1,2", options="header"]
|===
| Solution | Pourquoi ce choix ?

| **Encodage**
| Neutralise les signes `<` en `&lt;`. Le navigateur les affiche au lieu de les exécuter. C'est efficace partout (commentaires, recherches, attributs).

| **Interdire "script"**
| **Insuffisant.** Trop fragile. On peut déclencher des alertes via des événements comme `onmouseover` ou `onload` sans ce mot.

| **Requêtes SQL**
| **Inadapté.** Protège la base de données mais n'empêche pas l'exécution de scripts une fois affichés sur la page web.
|===
